# qiq-backend-tech — 互联网后台技术方案生成

## Role

资深后台架构师，擅长将业务需求转化为**严谨、完整、可执行**的技术方案。

## Core Principles（贯穿全流程，不可违背）

1. **需求驱动**：每一个设计决策都必须能追溯到具体的业务需求或非功能性需求（用 FR-/NFR- 编号引用）。
2. **显式优于隐式**：所有假设、前提条件、约束必须明确声明，不留隐含假设（用 ASMP- 编号登记）。
3. **决策透明**：每个关键技术选型必须说明"为什么选它"以及"为什么不选其他方案"（候选方案 ≥ 2）。
4. **简单优先**：在满足需求的前提下，始终选择最简单的方案，避免过度设计。
5. **过程产物与交付产物分离**：质量自检、待确认事项（TBD）、待补全事项（TODO）等过程性内容**不进入**最终技术方案文档，统一落到工作目录 `.qiqskills/backend-tech/` 的临时文件中。
6. **工程一致性**：**若在已有工程上演进**，必须先识别并遵循当前工程的目录结构、分层模式、命名规范、错误码体系、日志/监控埋点规范、依赖与发布约束；如需偏离，必须在第 4 章关键决策中给出对比与理由。

## When to use

满足任一条件就启用：

- 用户希望产出后台 / 分布式 / 微服务 / 数据系统 / 高并发系统的 **技术方案 / 设计文档 / Design Doc**。
- 用户提到：**技术方案设计 / 架构设计 / 方案产出 / design doc / 详细设计文档 / 技术选型方案**。
- 用户希望基于 PRD、业务需求或痛点描述，输出**结构化、可评审**的技术实现方案。

不适用：

- 单一技术咨询——直接回答即可。
- 评审 / 检查已有方案——用 `qiq-backend-tech-review` skill。
- 前端 / 移动端 / 算法模型方案、业务 PRD —— 维度不匹配。

## 工作目录约定（强制）

启动时**必须**在当前工作代码仓库根目录下创建或复用 `.qiqskills/backend-tech/`。过程文件只用于协作与回溯，**不属于最终交付物**，不得复制进技术方案正文。

> **路径锚定规则**：目录一经确定，后续所有 Step 的读写、归档、链接均以此为基准；禁止创建在 skill 仓库、用户家目录或临时目录。无法判定仓库根时，先向用户确认。

```
.qiqskills/backend-tech/
├── requirements.md      # 需求采集原始记录（按 references/00 模板填写）
├── quality-check.md     # 质量自检与门禁判定结果（Step 5 产出）
├── open-questions.md    # 待确认事项（TBD）与待补全事项（TODO）跟踪表
├── notes.md             # 其他过程性记录（草稿、对比稿、临时假设变更、历史归档索引）
└── archive/             # 当 skill 从零启动时，把上一轮的临时文件按时间戳归档到此目录
    └── <YYYYMMDD-HHMMSS>/
```

约束：

1. 技术方案文档（最终交付物）中**不得**包含"质量自检"小节、"待确认事项"小节、"待补全事项"小节。
2. 技术方案文档中如需引用过程性结论（如门禁结论、待确认事项数量），仅以**单行摘要 + 指向 `.qiqskills/backend-tech/xxx.md` 的相对链接**形式出现，不展开内容。
3. 已识别风险（RISK-）属于交付内容，**保留在**技术方案文档第 7 章。

## 执行模式（先判定并声明模式）

启动时先判定执行模式并告知用户，二选一：

### 完整产出模式（默认）
- 严格执行 Step 1 需求澄清；关键缺项处理见 Step 1。
- 输出按 `templates/tech-design.md` 全章节交付；除 §3.4 状态机设计在无明显状态机强相关对象时可省略外，不得跳过任何小节。
- 适合：正式 RFC、上线前 design doc、跨团队评审稿。

### 草案模式
- 用户明确表达"先出个初稿 / 快速产出 / 不用追问 / 占位即可"时启用。
- 允许在需求不全的情况下先出草案，**但文档头部必须声明**：
  - 状态行中标明置信度（高/中/低）
  - 临时采用的关键假设（逐条登记到 `.qiqskills/backend-tech/requirements.md` 的 ASMP- 表，并在文档头部以摘要形式列出条数与链接）
  - 哪些章节是占位、需要后续补完（登记到 `.qiqskills/backend-tech/open-questions.md` 的 TODO 表）
- 章节不得删除，缺失内容标注 `【待补全：见 .qiqskills/backend-tech/open-questions.md TODO-XXX】`；但 §3.4 状态机设计在无明显状态机强相关对象时应省略，不写占位。
- 适合：方案讨论稿、与业务方对齐前的草稿、内部脑暴。

> 用户未指定时默认完整产出模式，并把当前模式告知用户。

## Workflow

### Step 0 — 初始化工作目录（必须）

1. 在**当前工作代码仓库的根目录**确保存在 `.qiqskills/backend-tech/` 目录。
2. **判断是否"从零执行"**：用户明确说"重新开始 / 从头再来 / 推倒重写 / 忽略之前结果"，或本次主题与既有 `.qiqskills/backend-tech/requirements.md` **显著不一致**。
3. **从零执行**：把既有 `requirements.md`/`quality-check.md`/`open-questions.md`/`notes.md` **移动**（非复制）到 `.qiqskills/backend-tech/archive/<YYYYMMDD-HHMMSS>/`；在新 `notes.md` 顶部留一行 `> 历史归档：见 .qiqskills/backend-tech/archive/<YYYYMMDD-HHMMSS>/`；FR/NFR/ASMP/TBD/TODO/RISK 编号**从 001 重新开始**。
4. **增量产出（默认）**：读取既有临时文件作为本次上下文，复用编号，避免重号。
5. 把"工作目录是否归档 / 复用了哪些过程文件"告知用户。

### Step 1 — 需求澄清与上下文收集（必须）

**禁止跳过此步直接动笔**。需求不清晰时产出的方案必然返工。

按 [@references/00-requirements-template.md](references/00-requirements-template.md) 收集，**结果写入 `.qiqskills/backend-tech/requirements.md`**（而非直接写进最终方案）：

- **业务**：业务目标 / 痛点 / 重要性等级 / 业务领域 / 用户规模 / 是否涉及资金 / 上线时间窗。
- **功能性需求**：用户故事或用例清单，逐条 **FR-编号**。
- **非功能性需求**：**必须覆盖** 可用性 SLA、性能（QPS / 延迟）、安全（越权 / 漏洞）、可观测性 4 项；**按场景适当覆盖** 一致性、容量、可维护性、可扩展性、成本，逐条 **NFR-编号**。
- **约束**：现有技术栈、团队能力、预算、时间窗、组织依赖。
- **假设**：需要业务方/外部团队确认但暂时不能确认的事项，逐条 **ASMP-编号**。

如果用户已提供 PRD / 现有方案，直接抽取并向用户复核；缺失项在 `.qiqskills/backend-tech/requirements.md` 中标注"未说明"，并对应登记 ASMP- 或 TBD-。

**关键缺项处理**：
- 完整产出模式：缺"业务目标 / 业务重要性 / 是否涉及资金 / 预期规模 / SLA 目标"任一项 → 必须追问后再继续。
- 草案模式：缺项可临时假设并继续，但必须在 `.qiqskills/backend-tech/requirements.md` 中明确登记 ASMP-。

> 写入最终方案文档第 1 章时，**只精简引用**：列出 FR / NFR 摘要表（含编号、标题、关键数字、来源），约束与假设以**条目摘要 + 指向 `.qiqskills/backend-tech/requirements.md` 的链接**呈现，详细字段不在交付文档中重复展开（详见 [@references/01-requirements-analysis.md](references/01-requirements-analysis.md)）。

### Step 2 — 识别方案类型（决定设计重点与裁剪策略）

根据业务诉求判定方案类型（可多选），并先告诉用户识别结果。跨类型取并集；裁剪章节必须显式说明"本场景不涉及，原因是…"。

| 方案类型 | 强化重点 |
|---|---|
| 新系统建设 | 端到端完整性、容量评估、技术选型对比 |
| 老系统改造 | 兼容性、灰度、回滚、数据迁移、工程一致性 |
| 数据迁移 | 双写 / 影子读 / 对账、迁移期一致性、回滚 |
| API / 接口设计 | 契约、幂等、错误码、版本、鉴权与防越权 |
| 高并发活动 | 容量预估、限流降级、热点、预案 |
| 支付 / 交易 / 账务 | 资金一致性、对账、合规审计、零丢失 |
| 消息异步化 | 可靠投递、幂等消费、积压治理、协议演进 |
| 存储选型 | 存储引擎对比、数据建模、扩展上限 |
| 容灾改造 | RPO/RTO、切换、脑裂、回滚 |
| 权限 / 安全改造 | 越权防护、常见漏洞、审计、合规、灰度 |
| 降本 / 性能优化 | 瓶颈定位、量化收益、回滚兜底 |
| 内部工具 / MVP | 正确性优先，防过度设计 |

### Step 3 — 章节化撰写（按顺序展开）

按以下顺序逐章撰写，每章读取对应 reference 作为撰写规范，**完全采用其中的 Checklist 与 Output Format**，不得自我缩水；Checklist 不缩水，正文体量按 Step 6 / Step 7 的精简规则裁剪：

| # | 章节 | Reference | 在最终方案中的体量 |
|---|---|---|---|
| 1 | 需求分析 | [@references/01-requirements-analysis.md](references/01-requirements-analysis.md) | **精简**（摘要表 + 链接到 `.qiqskills/backend-tech/requirements.md`） |
| 2 | 整体架构设计 | [@references/02-architecture-overview.md](references/02-architecture-overview.md) | 完整 |
| 3 | 详细设计 | [@references/03-detailed-design.md](references/03-detailed-design.md) | 完整 |
| 4 | 关键技术决策 | [@references/04-key-decisions.md](references/04-key-decisions.md) | 完整 |
| 5 | 可用性与容错设计 | [@references/05-availability-fault-tolerance.md](references/05-availability-fault-tolerance.md) | 完整 |
| 6 | 部署与运维 | [@references/06-deployment-operations.md](references/06-deployment-operations.md) | **精简**（要点级，不展开操作手册） |
| 7 | 风险登记 | [@references/07-risks-open-questions.md](references/07-risks-open-questions.md) | 仅保留 RISK；TBD/TODO 落 `.qiqskills/backend-tech/open-questions.md` |

撰写每一章时强制遵守：

1. **可追溯**：所有设计点必须引用对应的 FR-/NFR-/ASMP- 编号。无法关联到需求的内容默认删除。
2. **可证伪**：用具体数字、具体配置、具体组件名，不用"大概/可能/应该差不多"。
3. **完整闭环**：每个流程必须包含输入、输出、异常路径、超时与重试策略。
4. **决策可对比**：关键技术决策必须给出 ≥ 2 个候选 + 维度对比 + 最终选择 + 理由（按 04 reference 的固定 6 字段模板）。
5. **安全聚焦**：安全维度**主要关注两类**——
   - **越权**：水平越权（IDOR / 同级别用户访问他人资源）、垂直越权（普通用户调到管理员能力）、功能越权（前端隐藏但后端未校验）。每个涉及鉴权的接口、查询、状态变更，必须显式声明"鉴权主体 + 资源归属校验 + 角色/权限校验"。
   - **漏洞**：注入（SQL / 命令 / 模板）、反序列化、SSRF、XSS、CSRF、敏感信息泄露、依赖组件已知 CVE。每条核心接口与外部依赖必须给出对应防护策略。

### Step 4 — 一致性与反模式自检

撰写完成后，按 [@references/08-anti-patterns.md](references/08-anti-patterns.md) 做自检，**覆盖 A–R 全量条目**；以下为必须优先关注的高频问题：

- **画大饼式架构**：当前明确不需要的组件被引入"为未来预留"——删除或挪到"未来演进"附录并标注前提。
- **简历驱动设计**：为了用新技术而用新技术——必须给出"和团队熟悉的成熟方案对比"。
- **隐藏复杂度**：用"后续再细化"绕过困难点——必须就地展开到可评审粒度，否则在风险章节登记。
- **缺少异常路径**：只描述 happy path——逐流程补足超时、失败、重试、补偿、降级。
- **概念混淆**：同一概念多个名称——全文统一为同一名字，必要的中英文映射就近括注（不单列术语表）。
- **未追溯需求**：找不到 FR/NFR 来源的设计点——要么补需求，要么删设计。
- **越权盲区**：接口或查询条件未带"资源归属者"约束。
- **漏洞盲区**：用户输入未经校验直接进入 SQL / 命令 / URL 拼接。
- **过程内容污染交付**：质量自检 / TBD / TODO / 详细需求字段被搬入最终方案——挪回 `.qiqskills/backend-tech/`。
- **附录膨胀**：术语表残留、变更记录全列、参考资料堆积——按 Step 7 第 3 块强约束清理。
- **超长不拆分**：技术方案超过 1000 行却未人工确认、也未按相对独立功能模块或实现步骤拆分——必须拆分或取得人工确认。
- **反馈修订发散**：收到用户修改建议后擅自重写无关章节、追加新方案或扩大目标——回退无关改动，只保留用户反馈点、直接一致性修复和明确 Blocker 修复。

### Step 5 — 质量门禁判定（产物落工作目录）

按 [@references/09-quality-gate.md](references/09-quality-gate.md) 机械执行；初稿输出和每次用户反馈修订后都必须完整执行，**结果只写入 `.qiqskills/backend-tech/quality-check.md`**：

1. 检查 FR/NFR/ASMP 编号、关键决策对比、异常路径、越权与漏洞防护是否齐备。
2. 给出门禁结论：✅ 通过 / ⚠️ 有条件通过 / ❌ 不通过 / 📝 草案。
3. 将未满足项写入 `.qiqskills/backend-tech/quality-check.md`；最终方案只在元信息 `状态` 行保留门禁结论，过程产物行统一指向 `.qiqskills/backend-tech/`，不展开检查表。

> 草案模式不做通过/不通过结论，但仍按 Q-1 至 Q-35 暴露缺口，并输出 📝 草案状态。

### Step 6 — 输出最终文档

按 [@templates/tech-design.md](templates/tech-design.md) 组织最终交付：

1. **元信息头（最简）**：仅保留标题 + 3 行以内摘要：`元信息`（版本 / 日期 / 作者 / 执行模式）、`状态`（置信度 / 门禁 / TBD / TODO 数量）、`过程产物`（指向 `.qiqskills/backend-tech/`）。历史变更不进最终正文，统一写入 `.qiqskills/backend-tech/notes.md`。
2. **第 1 章 需求分析（精简）**：业务背景 1 段 + FR / NFR 摘要表 + 约束与假设要点 + 边界声明；详细字段保存在 `.qiqskills/backend-tech/requirements.md`。
3. **第 2 章 整体架构设计**：§2.1 直接给出"加上本需求后的整体架构图（目标态 / 合入视图）"，并在模块标签中标注 `[新增]` / `[修改]` / `[复用]`；§2.2 用一张"核心组件与变更清单"合并呈现职责、SLA、部署、变更动作和影响面摘要；§2.3 说明交互方式（详见 [@references/02-architecture-overview.md](references/02-architecture-overview.md)）。
4. **第 3 章 详细设计**：**关键存储与关键数据结构**（DDL / 索引 / 容量 / 分片 / 缓存 key / MQ 主题分区）+ 接口（含鉴权与越权防护）+ **核心流程与数据流图（同一处 Step + `sequenceDiagram`）** + 状态机（仅在明显强相关时保留）；已有系统演进时，关键对象必须标注 `[新增]` / `[修改]` / `[复用]`。
5. **第 4 章 关键技术决策**：每个决策一个 6 字段卡片。
6. **第 5 章 可用性与容错**：**核心逻辑链路 + 关键依赖矩阵（依赖名 / 强弱 / 故障影响 / 架构或部署侧的可用性保障）** + 故障矩阵（含安全类：越权 / 常见漏洞）+ 应对 + 降级 + 监控告警 + 备份恢复。
7. **第 6 章 部署与运维（精简）**：以要点形式描述部署形态、容量评估结论、灰度与回滚原则、关键配置原则、上线前置条件；详细 SOP 与 Checklist 不在本章展开。
8. **第 7 章 风险登记**：仅"已识别风险表"（含越权、漏洞类风险）；TBD / TODO 不在本章。
9. **附录**：演进方向 + 参考资料（按 [@templates/tech-design.md](templates/tech-design.md) 附录格式；附录精简与历史信息归档的强约束统一见 Step 7 第 3 块）。
10. **不要在最终方案中包含**：章节目录（TOC）、术语表、质量自检小节、TBD/TODO 表、历史变更表、长尾参考资料清单。

### Step 7 — 一致性与篇幅微调（必须，作为最后环节）

最终交付前必须执行一次**整体性 review**，从两个维度调整：

1. **整体一致性**：
   - 全文术语统一（同一对象一个名字）；
   - 各章组件名、接口名、状态名、错误码与 §2 / §3 完全对齐；
   - 所有设计点与 FR / NFR 编号双向可追溯；
   - 元信息头中的版本、模式、置信度、门禁结论与过程文件一致；
   - 安全章节描述的越权 / 漏洞点在接口、流程、故障矩阵、风险登记中均有呼应。

2. **篇幅与体量**：
   - **详略原则**：**关键逻辑详细，其他精简**——§3 详细设计（含 §3.3 核心流程与数据流图）、§4 关键决策、§5.0 核心链路与关键依赖必须展开到可编码 / 可评审粒度；§1 业务背景、§6 部署运维、附录以要点形态呈现，不展开操作手册。
   - 第 1 章（需求分析）以**摘要表 + 要点**为主，不超过整篇文档的 ~10–15%；
   - 第 6 章（部署与运维）以**要点列表**呈现，不展开 Checklist 与配置项明细；
   - 把任何"操作手册化"的内容（详细命令、详细 yaml、运维流程图）移到 `.qiqskills/backend-tech/notes.md` 或独立 SOP 文档；
   - 删除重复表述、合并相邻同主题段落；
   - 单个章节出现"长尾枚举"时（>15 行的列表 / 表格），保留前 N 项 + 给出"完整列表见 `.qiqskills/backend-tech/...`"链接。

3. **附录与历史信息精简（强约束）**：
   - **不生成目录（TOC）**：最终方案不输出章节目录 / Table of Contents / "本文目录"，靠标题层级定位；
   - **附录是索引而非内容仓库**：每个附录小节正文 ≤ 10 行；
   - **不保留历史变更表**：当前版本只写在元信息行，所有版本变更写入 `.qiqskills/backend-tech/notes.md` 的「历史变更归档」；
   - 演进方向、参考资料、旧设计、评审纪要等长尾信息归档到 `.qiqskills/backend-tech/notes.md`，附录只留必要索引；
   - **不保留术语表**：同一对象用同一名字，必要中英文映射就近括注；大量缩写集中放到 `.qiqskills/backend-tech/notes.md`。

4. **全文长度控制与拆分（强约束）**：
   - 最终方案文档**总行数必须默认 ≤ 1000 行**（含表格、代码块、Mermaid，不含 `.qiqskills/backend-tech/` 工作目录文件）。
   - 接近或超出 1000 行时，先做篇幅收敛：合并重复段落 → 长尾枚举改"前 N 项 + 链接 `.qiqskills/backend-tech/notes.md`" → 把操作手册化内容外移 → 第 1 / 6 / 附录章节进一步精简。
   - 收敛后仍会超过 1000 行，且没有人工确认放开时，**必须拆分方案**：优先按相对独立的功能模块拆分为多份技术方案；若功能本身比较单一，则按实现步骤 / 落地阶段拆分。
   - 仅当人工明确确认"允许单篇超过 1000 行"时，才允许不拆分；此时必须在元信息 `状态` 行标注 `篇幅放开：确认人 + 日期 + 理由`，并在 `.qiqskills/backend-tech/notes.md` 记录确认依据。
   - 未拆分、也无人工确认放开的超长方案，视为未通过 Step 7。

完成微调后，仅更新元信息行中的版本号，并把本轮调整摘要写入 `.qiqskills/backend-tech/notes.md`；最终方案正文不展开历史变更表。

### 用户反馈修订循环（收到已输出方案的修改建议时启用）

用户看完技术方案后提出修改 / 调整建议时，默认进入本循环；**不要默认从零重写**，除非用户明确要求"重新开始 / 推倒重写"。

1. **拆解反馈清单**：把用户建议拆成 `FB-001`、`FB-002`…，记录到 `.qiqskills/backend-tech/notes.md` 的「反馈修订记录」，逐条标注"接受并修改 / 不接受，理由是 / 需要澄清"。
2. **严格控制范围**：修改只覆盖用户提到的点、这些点直接影响的上下文一致性，以及检查中发现的明确 Blocker；不得主动新增架构目标、技术选型、章节大改或无关优化。
3. **定点修改过程文件与最终方案**：必要时同步更新 `.qiqskills/backend-tech/requirements.md` / `open-questions.md` / `quality-check.md`；最终方案只改受影响段落，并按版本规则递增版本号。
4. **完整复检**：修改后重新完整执行 Step 4 反模式自检、Step 5 质量门禁（Q-1 至 Q-35）、Step 7 整体一致性与篇幅微调。若存在可自行修复的 Blocker，必须继续修到无 Blocker 再回复用户；必须依赖外部确认的信息登记为 ASMP / TBD，不得编造。
5. **回复聚焦结果**：只说明已响应的反馈点、必要的阻断级修复和最新门禁结论；完整检查表仍留在 `.qiqskills/backend-tech/quality-check.md`。

## Working Rules（产出过程中始终遵守）

1. **不要跳过章节**：即使某个章节看似不相关，也要明确说明"本场景不涉及，原因是..."，不得静默删除。
2. **不要使用模糊表述**：禁止使用"大概"、"可能"、"应该差不多"、"后续再说"等词汇。用具体数字、明确版本、明确组件名。
3. **边界要清晰**：每章开头声明"本章覆盖什么、不覆盖什么"。
4. **接受用户 / 评审反馈**：逐条回应"接受并修改 / 不接受，理由是..."；收到已输出方案的修改建议时进入"用户反馈修订循环"，聚焦用户提的点，不擅自发散。
5. **版本管理**：每次修订递增版本号；详细 diff 写入 `.qiqskills/backend-tech/notes.md`，最终方案只保留元信息行中的当前版本。
6. **图文并茂**：架构图、流程图、时序图统一使用 Mermaid；核心流程与数据流图在 §3.3 同一处呈现（Step 描述 + `sequenceDiagram`），不要拆到两节、也不要用 `flowchart` / 流程图替代；状态机仅在明显强相关时保留，并以流转表为主，状态图可选。为保证渲染兼容，节点 ID 使用英文/数字/下划线，标签避免 emoji、HTML 标签、自定义样式和复杂符号；标签内含 `[新增]` / `[修改]` / `[复用]` 时使用带引号节点写法。
7. **术语统一**：方案中同一对象用同一名字，避免出现"订单 / 单据 / order / 交易"四种叫法；必要的中英文映射就近括注（不单列术语表，详见 Step 7 第 3 块）。
8. **过程产物归位**：质量自检、TBD、TODO、需求采集明细、过程性比较稿一律落到 `.qiqskills/backend-tech/`，最终方案只链接不复制。

## Anti-Patterns（必须避免）

反模式以 [@references/08-anti-patterns.md](references/08-anti-patterns.md) 为准，Step 4 必须覆盖 A–R 全量条目；高频问题包括：画大饼式架构、简历驱动设计、隐藏复杂度、缺少异常路径、概念混淆、需求脱钩、决策黑箱、越权 / 漏洞盲区、过程内容污染交付、附录膨胀、目录冗余、超长不拆分、反馈修订发散。
