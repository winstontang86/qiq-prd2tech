# 第 1 章 需求分析 — 撰写规范（精简交付 + 详细工作目录）

## 章节目标

把模糊的业务诉求转化为 **可追溯、可验收、可证伪** 的需求清单。本章是后续所有设计的"锚"，所有设计点必须能反向追溯到本章的 FR / NFR / ASMP 编号。

## 双层产出（重要）

需求采集与撰写分两层，避免最终方案文档被冗长字段占满：

1. **过程层（详细）**：所有详细字段写入工作目录 `.qiqskills/backend-tech/requirements.md`，按 [@references/00-requirements-template.md](00-requirements-template.md) 全字段填写。这一层是评审/回溯的事实库。
2. **交付层（精简）**：在最终技术方案的"第 1 章 需求分析"中**只放摘要 + 关键数字 + 链接**，不重复展开。摘要要求：
   - 业务背景：1 段，≤ 5 句，含可量化目标。
   - FR 摘要表：编号 / 标题 / 优先级 / 验收关键指标（一行）。
   - NFR 摘要表：编号 / 维度 / 关键数字（一行）。
   - 约束与假设：仅以要点列出影响最大的 ≤ 5 条，其余条目链接到 `.tech-design/requirements.md`。
   - 边界声明：覆盖 / 不覆盖 / 上下游边界。

> 撰写时**先**完整填写 `.qiqskills/backend-tech/requirements.md`，**再**将摘要复制到方案第 1 章。绝不允许跳过详细字段直接写摘要。

## 工作目录详细字段要求

详细字段按下列规范填写到 `.qiqskills/backend-tech/requirements.md`：

### 业务背景与目标
- **3 句话以内** 讲清楚：当前在哪里 / 痛点是什么 / 期望到哪里。
- 必须包含可量化的业务目标（如"支付成功率从 99.5% 提升到 99.9%"），不接受"提升用户体验"之类的虚词。
- 引用业务方提供的数据、PRD 链接或会议纪要。

### 功能性需求清单（FR-）

每条需求**必须**包含：

| 字段 | 说明 |
|---|---|
| 编号 | `FR-001`、`FR-002`…按数字递增 |
| 标题 | 一句话功能描述 |
| 描述 | 用户故事或场景化描述（"作为 X，我希望 Y，以便 Z"） |
| 输入 / 输出 | 关键入参、出参 |
| 验收标准 | 如何验证需求被满足（具体操作 + 期望结果） |
| 优先级 | P0（必做）/ P1（应做）/ P2（可做） |
| 来源 | PRD 章节 / 业务方姓名 / 会议纪要 |

**示例**：

> **FR-003** 订单超时自动取消
> **描述**：作为商家，我希望未支付订单在 30 分钟后自动取消，以便释放库存。
> **输入**：订单创建后未支付。
> **输出**：订单状态变更为 `CANCELLED`，库存回滚 +N。
> **验收标准**：创建订单后不支付，30 分钟（±10s 容差）内库存恢复且订单状态正确。
> **优先级**：P0
> **来源**：PRD §3.2 / 业务方张三 2026-04-12

### 非功能性需求清单（NFR-）

**必须覆盖（强制 4 项）**：可用性 / 性能（QPS + 延迟）/ 安全（含越权 + 漏洞）/ 可观测性。
**按场景适当覆盖（5 项）**：一致性、容量、可维护性、可扩展性、成本。

每条 NFR 都要有具体数值（数字 / 等级 / 范围）；"未说明"的项必须显式写出并登记 ASMP-。

| 维度 | 必填? | 要求示例 |
|---|---|---|
| 可用性 SLA | **必** | 全年可用性 ≥ 99.95%；单次故障 MTTR ≤ 15 分钟 |
| 性能（QPS） | **必** | 峰值 10,000 QPS（写 2,000 + 读 8,000）；日均 5,500 QPS |
| 性能（延迟） | **必** | 核心接口 P99 ≤ 200ms，P999 ≤ 500ms（逐接口列出） |
| 安全（**重点**） | **必** | 越权防护：水平 / 垂直 / 功能越权全覆盖；漏洞防护：注入 / 反序列化 / SSRF / XSS / CSRF / 敏感信息泄露 / 依赖 CVE 全覆盖；审计日志保留 ≥ 90 天 |
| 可观测性 | **必** | 核心链路全量 Trace；错误日志 1 分钟内可检索；关键业务指标接入大盘 |
| 数据一致性 | 适当 | 资金类强一致；订单状态最终一致，窗口 ≤ 3s |
| 容量（数据量） | 适当 | 1 年内订单表 5 亿行，日增 150 万行；单行 ≈ 2KB |
| 容量（连接数） | 适当 | 长连接 10 万 / 单实例；DB 连接池 ≤ 200 |
| RPO / RTO | 适当 | RPO = 0（资金不丢失）；RTO ≤ 5 分钟（机房级故障） |
| 可维护性 | 适当 | 单服务部署 ≤ 10 分钟；回滚 ≤ 5 分钟；新成员上手 ≤ 2 周 |
| 可扩展性 | 适当 | 计算层无状态可水平扩展；存储层分片可在线扩容；接口向前兼容 ≥ 90 天 |
| 成本 | 适当 | 单笔交易边际成本 ≤ X 元；机器年预算 ≤ Y 万元；冗余系数 ≤ Z |

**示例**：

> **NFR-002** 性能：订单创建接口 P99 ≤ 150ms，峰值 1,000 QPS（来源：业务方 2026 大促预估，基线 400 QPS × 2.5 倍冗余）。
>
> **NFR-006** 安全：1) 任何按 ID 查询的接口必须做资源归属校验（防水平越权）；2) 所有用户输入参数走白名单 / 强类型校验（防注入）；3) 关键依赖每月扫描 CVE，高危 ≤ 7 天修复。

### 约束条件

逐条列出（不分编号也可，但要清晰）：

- **技术栈约束**：必须使用公司中间件 X / 不允许引入新语言 / 必须兼容 JDK 8。
- **团队约束**：可用人力 X 人 × Y 周；团队对 Z 技术不熟悉，需要培训成本。
- **时间窗约束**：必须在大促前 N 周上线；冻结期 [日期 a, 日期 b]。
- **预算约束**：机器预算 ≤ X 万元 / 年；不允许采购第三方 SaaS。
- **组织依赖**：DBA 排期需要 X 周；安全审计平均 Y 周。

### 核心假设清单（ASMP-）

**所有未确认但影响设计的事项必须登记**。每条包含：

| 字段 | 说明 |
|---|---|
| 编号 | `ASMP-001`、`ASMP-002`…按数字递增 |
| 描述 | 假设内容（如"假设 1 年后 DAU 不超过 500 万"） |
| 影响 | 假设不成立时影响哪些设计（如"D-3 数据库选型可能需要重选"） |
| 验证方式 | 何时、由谁、用什么手段验证 |
| 当前状态 | 待确认 / 已确认 / 已推翻 |

> 同样需要业务方决策但暂时未答的事项，应同时登记到 `.qiqskills/backend-tech/open-questions.md` 的 TBD 表中，避免遗漏。

## 反模式

- ❌ "提升用户体验" / "支持高并发" / "保证可用性" — 没有数字就是没有需求。
- ❌ FR 没有验收标准 — 验收时必扯皮。
- ❌ NFR 全靠拍脑袋 — 必须给出基线数据来源（业务方 / 历史监控 / 类比系统）。
- ❌ 假设不登记 — 假设不成立时无人负责。
- ❌ 把详细字段直接写进最终方案 — 第 1 章应当精简，让评审者快速对齐目标，详细字段去 `.qiqskills/backend-tech/requirements.md` 查。
- ❌ NFR 安全维度只写"等保三级"——必须显式覆盖越权与常见漏洞。
