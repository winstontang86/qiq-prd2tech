# 第 1 章 需求分析 — 撰写规范

## 章节目标

把模糊的业务诉求转化为 **可追溯、可验收、可证伪** 的需求清单。本章是后续所有设计的"锚"，所有设计点必须能反向追溯到本章的 FR / NFR / ASMP 编号。

## 必写小节

### 1.1 业务背景与目标
- **3 句话以内** 讲清楚：当前在哪里 / 痛点是什么 / 期望到哪里。
- 必须包含可量化的业务目标（如"支付成功率从 99.5% 提升到 99.9%"），不接受"提升用户体验"之类的虚词。
- 引用业务方提供的数据、PRD 链接或会议纪要。

### 1.2 功能性需求清单（FR-）

每条需求**必须**包含：

| 字段 | 说明 |
|---|---|
| 编号 | `FR-001`、`FR-002`…按数字递增 |
| 标题 | 一句话功能描述 |
| 描述 | 用户故事或场景化描述（"作为 X，我希望 Y，以便 Z"） |
| 输入 / 输出 | 关键入参、出参 |
| 验收标准 | 如何验证需求被满足（具体操作 + 期望结果） |
| 优先级 | P0（必做）/ P1（应做）/ P2（可做） |
| 来源 | PRD 章节 / 业务方姓名 / 会议纪要 |

**示例**：

> **FR-003** 订单超时自动取消
> **描述**：作为商家，我希望未支付订单在 30 分钟后自动取消，以便释放库存。
> **输入**：订单创建后未支付。
> **输出**：订单状态变更为 `CANCELLED`，库存回滚 +N。
> **验收标准**：创建订单后不支付，30 分钟（±10s 容差）内库存恢复且订单状态正确。
> **优先级**：P0
> **来源**：PRD §3.2 / 业务方张三 2026-04-12

### 1.3 非功能性需求清单（NFR-）

**必须** 至少覆盖以下维度，每个都要有具体数值。"未说明"的项必须显式写出并登记 ASMP-。

| 维度 | 要求示例 |
|---|---|
| 可用性 SLA | 全年可用性 ≥ 99.95%；单次故障 MTTR ≤ 15 分钟 |
| 性能（QPS） | 峰值 5,000 QPS（写 1,000 + 读 4,000）；日均 1,500 QPS |
| 性能（延迟） | 核心接口 P99 ≤ 200ms，P999 ≤ 500ms（逐接口列出） |
| 数据一致性 | 资金类强一致；订单状态最终一致，窗口 ≤ 3s |
| 容量（数据量） | 1 年内订单表 5 亿行，日增 150 万行；单行 ≈ 2KB |
| 容量（连接数） | 长连接 10 万 / 单实例；DB 连接池 ≤ 200 |
| RPO / RTO | RPO = 0（资金不丢失）；RTO ≤ 5 分钟（机房级故障） |
| 安全合规 | 等保三级；用户敏感数据加密落盘；审计日志保留 180 天 |
| 可观测性 | 核心链路全量 Trace；错误日志 1 分钟内可检索 |
| 可维护性 | 单服务部署 ≤ 10 分钟；回滚 ≤ 5 分钟 |

**示例**：

> **NFR-002** 性能：订单创建接口 P99 ≤ 150ms，峰值 1,000 QPS（来源：业务方 2026 大促预估，基线 400 QPS × 2.5 倍冗余）。

### 1.4 约束条件

逐条列出（不分编号也可，但要清晰）：

- **技术栈约束**：必须使用公司中间件 X / 不允许引入新语言 / 必须兼容 JDK 8。
- **团队约束**：可用人力 X 人 × Y 周；团队对 Z 技术不熟悉，需要培训成本。
- **时间窗约束**：必须在大促前 N 周上线；冻结期 [日期 a, 日期 b]。
- **预算约束**：机器预算 ≤ X 万元 / 年；不允许采购第三方 SaaS。
- **组织依赖**：DBA 排期需要 X 周；安全审计平均 Y 周。

### 1.5 核心假设清单（ASMP-）

**所有未确认但影响设计的事项必须登记**。每条包含：

| 字段 | 说明 |
|---|---|
| 编号 | `ASMP-001`、`ASMP-002`…按数字递增 |
| 描述 | 假设内容（如"假设 1 年后 DAU 不超过 500 万"） |
| 影响 | 假设不成立时影响哪些设计（如"D-3 数据库选型可能需要重选"） |
| 验证方式 | 何时、由谁、用什么手段验证 |
| 当前状态 | 待确认 / 已确认 / 已推翻 |

## Checklist（撰写完逐条勾选）

- [ ] 业务目标可量化，无虚词。
- [ ] FR 全部按统一字段写完，无"待补充"。
- [ ] NFR 至少覆盖 SLA / QPS / 延迟 / 一致性 / 容量 / 安全 6 项，且都有具体数值。
- [ ] 约束条件分技术 / 团队 / 时间 / 预算 / 组织 5 类列出。
- [ ] 所有未确认事项已登记 ASMP-，并指明影响哪些设计点。
- [ ] 每条 FR / NFR 都标了来源（PRD / 业务方 / 会议）。
- [ ] 后续章节的设计点都能追溯到本章某条编号。

## 反模式

- ❌ "提升用户体验" / "支持高并发" / "保证可用性" — 没有数字就是没有需求。
- ❌ FR 没有验收标准 — 验收时必扯皮。
- ❌ NFR 全靠拍脑袋 — 必须给出基线数据来源（业务方 / 历史监控 / 类比系统）。
- ❌ 假设不登记 — 假设不成立时无人负责。
