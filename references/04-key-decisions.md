# 第 4 章 关键技术决策 — 撰写规范

## 章节目标

让评审者看清"**为什么是这个选择，而不是别的**"。每个关键决策必须以固定的 6 字段卡片输出，禁止"我觉得"式陈述。

## 哪些算"关键决策"？（识别清单）

任一命中即必须立卡：

- 存储选型（MySQL vs PostgreSQL vs TiDB；Redis vs MemoryDB；MongoDB vs HBase 等）。
- 消息中间件选型（Kafka vs RocketMQ vs RabbitMQ vs Pulsar）。
- 一致性模型（强一致 / 最终一致 / 因果一致）。
- 同步 vs 异步 通信方式。
- 数据分片策略（按用户 / 按时间 / 按业务键 / 一致性哈希）。
- 锁策略（DB 行锁 / 分布式锁 / 乐观锁 / 无锁 CAS）。
- 缓存策略（Cache Aside / Read Through / Write Through / Write Back）。
- 部署架构（单机房 / 同城双活 / 异地多活）。
- 鉴权方案（Session / JWT / OAuth2 / mTLS）。
- 自研 vs 开源 vs 公司平台中间件。
- 任何引入"新技术"的决策（团队首次使用）。

## 决策卡片模板（6 字段，**全字段必填**）

```
### 决策 D-N：<决策点标题>

**决策点**：要解决什么问题？关联的需求编号（FR-/NFR-/ASMP-）。

**可选方案**（≥ 2 个）：
- 方案 A：<名称 + 一句话描述>
- 方案 B：<名称 + 一句话描述>
- 方案 C（如有）：<名称 + 一句话描述>

**对比分析**（用表格，至少覆盖以下维度）：

| 维度 | 方案 A | 方案 B | 方案 C |
|---|---|---|---|
| 满足需求度（对应 FR/NFR） | | | |
| 实现复杂度 | | | |
| 性能（具体数字） | | | |
| 可用性 / 容灾 | | | |
| 可维护性 | | | |
| 团队熟悉度 | | | |
| 引入成本（机器/许可/学习） | | | |
| 长期演进空间 | | | |

**最终选择**：<选择哪一个>

**理由**：
1. <数据驱动的理由 1，引用对比表中的具体维度>
2. <数据驱动的理由 2>
3. <数据驱动的理由 3>

**前提与失效条件**：
- 当 <条件> 不再成立时，本决策需要重新评估。
- 例如："当 QPS 超过 50,000 时，方案 A 无法横向扩展，需切换到方案 C"。

**对应需求**：FR-XXX、NFR-XXX、ASMP-XXX
```

## 撰写规则

1. **候选方案 ≥ 2**：只写一个方案 = 没有决策。如果客观上只有一个合理选择，仍需列出"被排除的明显选项"并说明排除原因。
2. **对比维度具体化**：禁止"性能好 / 性能一般"。要写具体数字（如"单机 5w QPS / 单机 8k QPS"）或具体场景（如"原生支持顺序消息 / 需自建分区路由"）。
3. **理由数据化**：每条理由要么引用对比表中的格子，要么引用 FR/NFR 编号。禁止"它更稳定" / "团队更喜欢"等无依据陈述。
4. **失效条件必填**：每个决策都有适用边界。当输入参数（QPS、数据量、团队规模）变化到某阈值时，决策可能不再最优。这一阈值必须显式写出。
5. **诚实面对劣势**：被选方案的缺点必须明确写出，并在第 5 章（容错）或第 7 章（风险）中给出补救措施。

## 完整示例

```
### 决策 D-2：异步消息中间件选型

**决策点**：订单创建后需要异步通知库存、积分、推送等下游系统，需选择一款消息中间件。
关联：FR-007（订单状态顺序投递）、NFR-005（异步解耦削峰）、NFR-006（消息零丢失）。

**可选方案**：
- 方案 A：Apache Kafka（自建集群）
- 方案 B：Apache RocketMQ（公司中间件平台已托管）
- 方案 C：RabbitMQ（自建集群）

**对比分析**：

| 维度 | Kafka | RocketMQ | RabbitMQ |
|---|---|---|---|
| 满足 FR-007（顺序消息） | 需自建分区路由 + 消费幂等 | 原生 OrderConsumer 支持 | 单队列单消费者方可有序 |
| 满足 NFR-006（零丢失） | acks=all + min.isr=2 可达 | 同步双写 + 同步刷盘可达 | 镜像队列 + publisher confirm |
| 单机吞吐 | 50w msg/s | 10w msg/s | 1w msg/s |
| 实现复杂度 | 高（自建运维） | 低（平台托管） | 中（自建运维） |
| 团队熟悉度 | 2/8 人有经验 | 6/8 人有经验 | 0/8 人有经验 |
| 公司基础设施 | 无 | 已托管 | 无 |
| 引入成本 | 自建机器 +6 台 + 培训 | 0（接平台 API） | 自建机器 +4 台 + 培训 |
| 监控告警 | 自建 | 平台已提供 | 自建 |

**最终选择**：方案 B — RocketMQ（接公司中间件平台）

**理由**：
1. **顺序消息原生支持**：FR-007 要求订单状态变更按 orderId 顺序消费，RocketMQ OrderConsumer 开箱即用；Kafka 需自建路由层，开发周期 +2 周。
2. **零运维成本**：公司中间件平台已托管，团队 0 运维投入；Kafka/RabbitMQ 均需自建，对应 +6 台机器与 1 人长期运维。
3. **团队熟悉度**：6/8 成员有 1 年以上 RocketMQ 经验，培训成本 0；Kafka 仅 2/8 熟悉，RabbitMQ 0/8。
4. **当前 QPS 1k 远低于 RocketMQ 单机 10w 上限**，性能充裕。

**前提与失效条件**：
- 当全局消息 QPS > 50,000 时，RocketMQ 单机吞吐成为瓶颈，需重新评估 Kafka。
- 当公司中间件平台 SLA 不再满足 NFR-006 时，需评估自建。
- 当出现跨地域多活诉求时，需重新评估 Pulsar / Kafka MirrorMaker 等方案。

**对应需求**：FR-007、NFR-005、NFR-006
```

## Checklist

- [ ] 已识别所有"关键决策"（用上方清单逐条比对）。
- [ ] 每个决策都用 6 字段模板，无字段缺失。
- [ ] 候选方案 ≥ 2，且每个候选都给出一句话描述。
- [ ] 对比维度至少 6 个，每格具体到数字或具体场景。
- [ ] 理由都引用对比表或 FR/NFR 编号，无"我觉得"。
- [ ] 失效条件已写明阈值。
- [ ] 被选方案的缺点已诚实写出，并在第 5 / 7 章给出对应补救。
- [ ] 决策编号（D-1, D-2…）连续无断号。

## 反模式

- ❌ "我们用 Kafka，因为大家都用" — 无对比、无数据。
- ❌ 候选方案只写一个 — 不是决策，是结论。
- ❌ "性能更好"未给数字 — 不可证伪。
- ❌ 不写失效条件 — 决策永远"对"，但实际并非如此。
- ❌ 隐藏被选方案的劣势 — 评审者发现后信任度归零。
