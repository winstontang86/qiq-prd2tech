# 第 5 章 可用性与容错设计 — 撰写规范

## 章节目标

证明系统在各类故障场景下仍能 **满足 NFR 中声明的 SLA、RPO、RTO**。所有故障必须显式枚举并给出应对策略，禁止"应该没问题"。

## 必写小节

### 5.1 故障场景枚举（故障矩阵）

**至少**覆盖以下场景，逐项判定"是否涉及"，不涉及需写明理由：

| 场景类别 | 具体场景 | 是否涉及 | 应对策略 | 引用编号 |
|---|---|---|---|---|
| 单机故障 | 业务实例 OOM / Crash | | 多副本 + K8s 重启 + LB 摘除 | FT-1 |
| 单机故障 | 单台 DB 主库宕机 | | 半同步复制 + MHA 切换 ≤ 30s | FT-2 |
| 单机故障 | 单台缓存节点宕机 | | Redis Cluster 自动 failover | FT-3 |
| 机房故障 | 单机房整体不可用 | | 同城双活 + DNS / 流量调度切换 | FT-4 |
| 网络分区 | 跨机房网络抖动 | | 调用方超时 + 熔断 + 本地降级 | FT-5 |
| 网络分区 | 客户端到网关丢包 | | 客户端重试 + 幂等保护 | FT-6 |
| 时钟漂移 | 节点时间偏差 > 1s | | NTP + 业务侧使用单调时钟 / Snowflake 容忍 | FT-7 |
| 依赖故障 | 强依赖下游服务宕机 | | 熔断 + 快速失败 + 降级响应 | FT-8 |
| 依赖故障 | 弱依赖下游服务宕机 | | 熔断 + 默认值降级 + 异步补偿 | FT-9 |
| 流量异常 | 突发流量超出预估 | | 限流（用户/接口/全局）+ 排队 + 自动扩容 | FT-10 |
| 流量异常 | 慢查询拖垮 DB | | SQL 审核 + 慢查杀手 + 读写分离 | FT-11 |
| 数据故障 | 误删 / 误更新 | | DDL 审核 + 闪回 + 全量备份 + binlog | FT-12 |
| 数据故障 | 数据不一致（脑裂） | | 仲裁机制 + 对账任务 + 告警 | FT-13 |
| 安全故障 | DDoS / CC 攻击 | | WAF + 限流 + 黑名单 | FT-14 |
| 发布故障 | 新版本 Bug 触发雪崩 | | 灰度 + 自动回滚 + 熔断 | FT-15 |

每条 **FT-编号** 应能在第 6 章（部署运维）或下文 5.2/5.3 中找到对应实现。

### 5.2 应对策略详细说明

对**每个 SLA 关键** 的故障场景，给出可落地的实现细节：

**示例 FT-2 — DB 主库宕机**：

- **检测**：MHA / Orchestrator 通过心跳，5s 内识别。
- **切换**：选举半同步复制最新的从库，绑定 VIP 切换。
- **超时**：切换 ≤ 30s（满足 NFR-008 RTO ≤ 60s）。
- **数据保证**：半同步保证 RPO ≈ 0；极端情况丢失最多 1 个事务（已纳入 ASMP-007）。
- **业务侧配合**：DAO 层短连接重试 + 指数退避；幂等接口 5 次重试。
- **演练**：每季度一次主从切换演练。

**示例 FT-8 — 强依赖下游宕机**：

- **熔断**：Sentinel / Hystrix；窗口 10s，错误率 > 50% 触发，半开 30s。
- **降级**：同步调用降级为返回固定错误码 + 引导客户端重试（不返回脏数据）。
- **隔离**：依赖按业务维度线程池隔离，避免拖死整个服务。
- **告警**：熔断触发即告警；降级期间 5 分钟未恢复升级到 P1。

### 5.3 降级策略

**列出每个可降级功能**，给出：

| 功能 | 触发条件 | 降级行为 | 影响范围 | 恢复方式 |
|---|---|---|---|---|
| 商品推荐 | 推荐服务熔断或 P99 > 1s | 返回静态热门列表 | 推荐准确度↓，CTR 预计 -10% | 推荐服务恢复后自动切回 |
| 实时评分 | 评分服务不可用 | 返回 T-1 离线评分 | 评分时效性 T-1 | 实时服务恢复后自动切回 |
| 订单创建（核心） | — | 不可降级（资金类） | — | — |

不可降级的功能必须显式列出。

### 5.4 监控与告警设计

**指标分层**（USE / RED / Golden Signals）：

| 层级 | 指标 | 阈值 | 告警等级 |
|---|---|---|---|
| 业务 | 订单创建成功率 | < 99.9%（5 分钟） | P1 |
| 业务 | 支付成功率 | < 99.5%（5 分钟） | P0 |
| 接口 | P99 延迟 | > 300ms（5 分钟） | P2 |
| 接口 | 错误率（5xx） | > 1%（1 分钟） | P1 |
| 系统 | CPU 使用率 | > 80%（5 分钟） | P2 |
| 系统 | 内存使用率 | > 85%（5 分钟） | P2 |
| DB | 主从延迟 | > 1s | P2 |
| DB | 慢查询数 | > 10 / 分钟 | P2 |
| 中间件 | MQ 消费积压 | > 10000 条或延迟 > 5 分钟 | P1 |
| 中间件 | 缓存命中率 | < 90%（10 分钟） | P3 |

**告警通道**：
- P0：电话 + 短信 + IM，5 分钟无响应升级到 leader。
- P1：短信 + IM，15 分钟无响应升级。
- P2：IM，半小时无响应升级。
- P3：IM 群通知。

**Trace / Log / Metric 三位一体**：
- 全链路 Trace 采样率：核心链路 100%，其余 1%。
- 错误日志结构化（JSON）+ 集中检索。
- 关键业务指标接入大盘，可下钻到实例。

### 5.5 数据备份与恢复策略

| 数据类型 | 备份策略 | 保留期 | 恢复演练频率 | 满足的 RPO/RTO |
|---|---|---|---|---|
| 订单 DB | 全量每日 + binlog 实时 | 全量 30 天，binlog 7 天 | 每季度 | RPO ≈ 0，RTO ≤ 30 分钟 |
| 用户 DB | 全量每日 + binlog 实时 | 同上 | 每季度 | 同上 |
| Redis 缓存 | RDB 每小时 + AOF | 7 天 | 每半年 | RPO ≤ 1h，RTO ≤ 15 分钟 |
| 对象存储 | 跨机房同步 | 永久 | 每年 | RPO ≈ 0，RTO ≤ 1h |

**恢复路径必须演练过**：演练记录、恢复时长、问题与改进项必须存档。

## Checklist

- [ ] 故障矩阵覆盖单机 / 机房 / 网络 / 时钟 / 依赖 / 流量 / 数据 / 安全 / 发布 9 类，每类至少 1 条。
- [ ] 每个"涉及"的故障都有 FT- 编号 + 应对策略 + 是否满足 NFR 的判定。
- [ ] SLA 关键故障的应对有可落地实现细节（检测方法、切换动作、超时数字）。
- [ ] 降级策略表覆盖所有可降级功能，并显式列出"不可降级"项。
- [ ] 监控告警指标按业务 / 接口 / 系统 / DB / 中间件分层，每条有阈值与等级。
- [ ] 告警分级 P0~P3 有明确通道与升级机制。
- [ ] 备份策略含全量 + 增量 + 保留期 + 恢复演练频率。
- [ ] RPO / RTO 数字与第 1 章 NFR 一致。

## 反模式

- ❌ "我们有多副本所以高可用" — 没有故障矩阵 = 没有论证。
- ❌ "建议加监控" — 监控就是设计的一部分，不是建议。
- ❌ 降级策略全靠口头 — 必须列出触发条件 + 降级行为 + 影响。
- ❌ RPO/RTO 数字与 NFR 不一致 — 自相矛盾。
- ❌ 备份从未演练 — 需要时多半恢复不出来。
- ❌ 监控阈值靠拍脑袋 — 必须基于历史数据或压测。
