# 第 6 章 部署与运维 — 撰写规范

## 章节目标

让 SRE / 运维同学看清"系统怎么部署、怎么扩缩、怎么发布、怎么改配置"。本章是"上线交接清单"。

## 必写小节

### 6.1 部署架构

#### 6.1.1 拓扑

字段化描述：

| 项 | 说明 |
|---|---|
| 部署形态 | K8s 容器 / 物理机 / VM / Serverless |
| 机房分布 | 单机房 / 同城双活 / 异地多活；机房名称：北京-A、北京-B |
| 实例规模 | 业务服务 6 副本（每机房 3）；DB 1 主 2 从（跨机房） |
| 资源规格 | 业务实例：4C8G；DB：32C128G + 4TB SSD |
| 网络拓扑 | VPC、安全组、跨机房专线带宽、入口与出口公网 |
| 流量入口 | LB → API 网关 → 业务服务 |

如果跨机房，必须画一张 **部署拓扑图**（Mermaid / PlantUML），标明：
- 各组件实例数与机房归属。
- 跨机房同步链路（DB 半同步、Redis 同步、MQ Mirror）。
- 流量路由策略（就近接入 / 单元化路由 / 主备切换）。

#### 6.1.2 容量评估

按 NFR 中的 QPS 和资源用量，反推实例数：

```
单实例处理能力（压测得出）：800 QPS（CPU 60%）
峰值 QPS：5000
冗余系数：1.5（单机房挂 1 副本仍能扛住）
单机房副本数 = ceil(5000 / 800 × 1.5) = 10
双机房总副本数 = 10 × 2 = 20
```

### 6.2 扩缩容策略

| 维度 | 策略 |
|---|---|
| HPA 触发指标 | CPU > 60% **或** P99 延迟 > 200ms 持续 3 分钟 |
| 扩容步长 | 每次 +30%（最少 +1） |
| 扩容上限 | 单机房 30 副本（受 DB 连接池上限限制） |
| 缩容触发 | CPU < 30% **且** P99 < 100ms 持续 15 分钟 |
| 缩容步长 | 每次 -1 副本，间隔 ≥ 5 分钟 |
| 预热 | 新实例启动后 60s 内不接入正式流量（grace period） |
| 大促前固定容量 | 关闭 HPA，固定到上限的 80% |

DB / 缓存 / MQ 的扩容策略也要写明（通常是垂直扩容 + 主动分片调整）。

### 6.3 发布策略

| 项 | 说明 |
|---|---|
| 发布方式 | 滚动发布 / 蓝绿 / 金丝雀（灰度） |
| 灰度比例 | 1% → 10% → 50% → 100%，每阶段观察 ≥ 30 分钟 |
| 灰度维度 | 按用户 ID 哈希 / 按地域 / 按内部白名单 |
| 健康检查 | 就绪检查 + 存活检查；新实例 60s 内未就绪则失败 |
| 自动回滚触发 | 错误率 > 1% **或** P99 > 300ms 持续 5 分钟 |
| 回滚时间 | ≤ 5 分钟（满足 NFR-XXX） |
| 数据库变更 | 通过平台审核；禁止破坏性 DDL；大表 DDL 走 gh-ost |
| 配置变更 | 走配置中心审核流程；高危配置（限流阈值、熔断开关）需双人审核 |
| 发布窗口 | 工作日 10:00–18:00；冻结期：大促前 1 周 + 节假日 |

**破坏性变更兼容协议**（接口或消息格式变更必须满足）：

1. 兼容期 ≥ 90 天：新旧字段并存。
2. 旧字段移除前 30 天：监控旧字段使用方，主动告警提醒。
3. 移除：所有调用方迁移完成后再下线。

### 6.4 配置管理

| 项 | 说明 |
|---|---|
| 配置存储 | 配置中心（如 Apollo / Nacos） |
| 配置分级 | 应用级 / 环境级（dev/test/staging/prod） |
| 加密配置 | DB 密码、第三方 token 走 KMS / Vault，不落明文 |
| 灰度配置 | 支持按实例 / IP / 标签灰度生效 |
| 变更审核 | 高危配置（限流、熔断、开关）双人审核 + 灰度生效 |
| 变更审计 | 所有配置变更记录 who / when / before / after，保留 ≥ 1 年 |
| 回滚 | 一键回滚到上一版本，≤ 1 分钟生效 |
| 默认值与兜底 | 配置拉取失败时使用本地默认值，避免启动失败 |

**关键配置清单**（列出所有"高危配置"）：

| 配置 Key | 含义 | 默认值 | 高危等级 | 修改流程 |
|---|---|---|---|---|
| `rate_limit.qps` | 全局限流阈值 | 5000 | 高 | 双人审核 + 灰度 |
| `circuit_breaker.error_rate` | 熔断错误率阈值 | 0.5 | 高 | 双人审核 |
| `feature.recommend_enabled` | 推荐功能开关 | true | 中 | 单人审核 |

### 6.5 上线 Checklist（建议附录）

- [ ] 上线 Checklist 完备。

## 反模式

- ❌ "用 K8s 部署" — 没有副本数、规格、机房分布等于没说。
- ❌ HPA 阈值随便填 — 必须基于压测数据。
- ❌ "灰度发布" 不写比例与时长 — 实际上线时无据可依。
- ❌ 配置散落在 yaml / 环境变量 / DB — 难以审计与回滚。
- ❌ 不写回滚方案 — 出事时只能硬扛。
- ❌ 数据库变更直接 ALTER 大表 — 锁表雪崩。
