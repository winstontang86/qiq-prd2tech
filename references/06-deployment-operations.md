# 第 6 章 部署与运维 — 撰写规范（要点级，不展开操作手册）

## 章节目标

让评审者在 **5 分钟内** 看清"系统怎么部署、容量是否合理、灰度与回滚原则是否到位、关键配置是否有兜底"。本章不是 SOP，不是操作手册，**不展开具体命令、详细 yaml、Checklist 全集**。

## 总原则

1. **要点优先，操作手册靠边**：本章在最终方案中应是 **1–2 页的要点**，不是 5+ 页的运维细则。
2. **关键判定可量化**：副本数、灰度比例、回滚时长、自动回滚阈值 必须给具体数字，不要"按需调整"。
3. **详细 SOP / 配置项明细 / 上线 Checklist 全集**：放入工作目录 `.qiqskills/backend-tech/notes.md` 或独立 SOP 文档；本章只点要害。

## 必写要点（4–5 项即止）

### 6.1 部署形态

只用一个小表说清：

| 项 | 内容 |
|---|---|
| 部署模式 | K8s 容器 / VM / Serverless；单机房 / 同城双活 / 异地多活 |
| 实例规模 | 业务服务 X 副本（每机房 Y）；DB 主从拓扑 |
| 资源规格 | 业务实例：xCxG；DB：xCxG + xTB SSD |
| 流量入口 | LB → 网关 → 业务服务 |

> 跨机房可附 1 张 ≤ 10 节点的部署拓扑图（Mermaid），重点标明：实例机房归属、跨机房同步链路、流量路由策略。**详细网络拓扑、安全组、专线配置不在本章**。

### 6.2 容量评估结论

只放结论性公式 + 数字，1 段 ≤ 6 行：

```
单实例处理能力：xxx QPS（CPU 60%，压测得出）
峰值 QPS：xxx
冗余系数：1.5
单机房副本数 = ceil(峰值 / 单实例 × 冗余) = N
```

### 6.3 发布与回滚原则

只列"判定标准"，不写完整发布流程：

| 项 | 关键数字 |
|---|---|
| 发布方式 | 灰度（金丝雀） |
| 灰度阶梯 | 1% → 10% → 50% → 100%，每阶段 ≥ 30 分钟 |
| 自动回滚触发 | 错误率 > 1% **或** P99 > 300ms 持续 5 分钟 |
| 回滚目标时间 | ≤ 5 分钟 |
| 破坏性变更兼容期 | ≥ 90 天，下线前 30 天监控告警 |

### 6.4 关键配置原则

仅 3–5 行要点：

- 配置统一存放于配置中心（如 Apollo / Nacos）。
- 高危配置（限流阈值、熔断开关、KMS 密钥、敏感开关）走 **双人审核 + 灰度生效**。
- 敏感配置（DB 密码、第三方 token）走 KMS / Vault，不落明文，不进 Git。
- 配置变更全量审计（who / when / before / after），保留 ≥ 1 年。
- 配置拉取失败有本地默认值兜底，不能阻塞启动。

> **高危配置完整清单**（Key / 默认值 / 修改流程） 放入 `.qiqskills/backend-tech/notes.md`，不在本章展开。

### 6.5 上线前置条件（要点）

不展开为 Checklist；列必须达成的"门"：

- 监控告警：业务 / 接口 / 系统 / DB / 中间件 / **安全（鉴权失败、WAF 命中、越权审计）** 全部分层配置并联调。
- 容量压测：达成 NFR 中 QPS / 延迟目标。
- 故障演练：第 5 章故障矩阵中标记"涉及"的场景已演练 ≥ 1 次。
- 备份恢复：演练通过，RPO / RTO 与 NFR 一致。
- 安全扫描：覆盖**越权（水平 / 垂直 / 功能）+ 常见漏洞（注入 / 反序列化 / SSRF / XSS / CSRF / 依赖 CVE）**，高危项已修复。
- 上下游兼容性：契约测试通过。
