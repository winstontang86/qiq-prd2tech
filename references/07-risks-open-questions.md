## 最终方案中保留的内容

### 7.1 已识别风险及缓解措施（RISK-）

每条风险**必须**包含：

| 字段 | 说明 |
|---|---|
| 编号 | `RISK-001`、`RISK-002`… |
| 类别 | 业务 / 技术 / 数据 / 安全（**含越权 + 漏洞**） / 合规 / 团队 / 时间 |
| 描述 | 一句话风险陈述（"如果 X，则会 Y"） |
| 触发概率 | 高 / 中 / 低（基于历史数据 / 经验判断） |
| 影响等级 | 高 / 中 / 低（金额 / 用户数 / 故障时长） |
| 风险等级 | 概率 × 影响 → 红 / 黄 / 绿 |
| 缓解措施 | 当前方案中已采取的对策（指向 FT- / D- 编号） |
| 残余风险 | 缓解后仍存在的风险（是否可接受） |
| 责任人 | 风险跟踪 owner |

**示例（技术）**：

> **RISK-002**
> **类别**：技术
> **描述**：RocketMQ 顺序消费在单机降级或扩容时，可能短暂破坏顺序性。
> **触发概率**：低（年均 1–2 次扩缩容）
> **影响等级**：中（可能出现订单状态短暂错乱，业务需对账修复）
> **风险等级**：黄
> **缓解措施**：1) 启用对账任务每 10 分钟扫描订单状态一致性（FT-13）；2) 消费侧基于 orderId + version 做幂等（D-3）。
> **残余风险**：扩缩容期间 ≤ 5 分钟可能出现状态错乱，对账后可恢复，业务方接受。
> **责任人**：张三

**示例（安全 - 越权）**：

> **RISK-005**
> **类别**：安全（越权）
> **描述**：订单查询接口 `GET /v1/orders/{orderNo}` 若仅依赖 URL 参数解析用户，存在水平越权（IDOR）风险。
> **触发概率**：中（历史同类系统年均出现 1 次）
> **影响等级**：高（涉及用户隐私 + 合规）
> **风险等级**：红
> **缓解措施**：1) 服务端二次校验 `order.user_id == token.userId`（D-7）；2) 鉴权失败审计日志 + 5 分钟告警阈值（5.4 监控）。
> **残余风险**：第三方 token 泄露场景仍可能被利用；缓解为 token 短 TTL + 异常 IP 风控。
> **责任人**：李四

**示例（安全 - 漏洞）**：

> **RISK-006**
> **类别**：安全（漏洞）
> **描述**：消息消费链路使用 Java 反序列化，存在已知反序列化攻击面。
> **触发概率**：低
> **影响等级**：高（RCE）
> **风险等级**：红
> **缓解措施**：1) 切换到 protobuf / JSON（D-9）；2) 反序列化白名单 + 类加载限制；3) 依赖每月 CVE 扫描。
> **残余风险**：白名单维护不及时风险；指定 owner 月度复审。
> **责任人**：王五

## 撰写规则

1. **风险不许写"已规避"**：所有风险都有残余概率，必须给出缓解措施 + 残余风险描述。
2. **不许把高风险藏成中风险**：等级判定需基于具体数字（如金额 / 影响用户数）而非主观意愿。
3. **TBD / TODO 不进交付文档**：所有 TBD / TODO 一律落 `.qiqskills/backend-tech/open-questions.md`，技术方案只在元信息头给条数摘要。
4. **TBD 必须有期望时间**：没有 deadline 的待确认事项 = 永远不会确认。
5. **责任人必须实名**：不能写"团队 / 项目组" — 没人负责 = 不会跟进。
6. **风险类别要平衡**：仅识别技术风险通常意味着没识全；业务 / 合规 / 团队 / **安全（越权 + 漏洞）** 风险必须独立审视。

## 风险识别参考清单（用于自检是否有遗漏）

逐条对照，命中的必须登记到最终方案 7.1（涉及未确认者同时登记到 `.qiqskills/backend-tech/open-questions.md` 的 TBD）：

- [ ] **业务风险**：需求变更概率？是否有竞品上线压力？
- [ ] **数据风险**：是否涉及敏感数据？数据迁移是否有丢失风险？
- [ ] **性能风险**：QPS 预估是否准确？是否有热点 key？
- [ ] **可用性风险**：是否依赖单点？跨机房是否有真实演练？
- [ ] **安全 - 越权风险**：所有按 ID 查询 / 状态变更 / 文件下载接口是否做了资源归属校验？管理端接口是否有角色校验？前端隐藏的入口后端是否独立校验？
- [ ] **安全 - 漏洞风险**：注入（SQL / 命令 / 模板 / LDAP）/ 反序列化 / SSRF / XSS / CSRF / 文件上传 / 路径穿越 / 敏感信息泄露 / 依赖组件 CVE 是否覆盖？
- [ ] **合规风险**：是否触及个保 / 等保 / 跨境数据？
- [ ] **第三方依赖**：第三方 SLA 是否覆盖？降级方案是否充分？
- [ ] **团队风险**：核心成员是否单点？是否有培训成本？
- [ ] **时间风险**：里程碑是否可达？是否有外部依赖排期不确定？
- [ ] **预算风险**：机器 / 第三方 / 人力是否超预算？

## 反模式

- ❌ "目前未发现风险" — 几乎不可能；通常意味着没认真识别。
- ❌ 把 TBD / TODO 直接塞进交付方案 — 污染交付文档，干扰读者重点。
- ❌ TBD 写"找时间确认" — 没 deadline 不会发生。
- ❌ 风险责任人写"团队" — 责任分散等于无责任。
- ❌ 风险等级靠感觉 — 必须是概率 × 影响的二维判定。
- ❌ 缓解措施与第 5 章脱钩 — 在 RISK 里说"会监控"但第 5 章没监控指标 = 假大空。
- ❌ 安全风险只有"DDoS" — 漏掉越权与代码漏洞，是大坑。
