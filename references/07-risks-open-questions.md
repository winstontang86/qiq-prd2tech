# 第 7 章 风险与待确认事项 — 撰写规范

## 章节目标

把所有"暂时无法在方案中解决、需要业务方/外部团队确认或在执行中持续观察"的事项 **显式登记**。这一章是项目交付后回顾时的"为什么没做到"的依据。

## 必写小节

### 7.1 已识别风险及缓解措施（RISK-）

每条风险**必须**包含：

| 字段 | 说明 |
|---|---|
| 编号 | `RISK-001`、`RISK-002`… |
| 类别 | 业务 / 技术 / 数据 / 安全 / 合规 / 团队 / 时间 |
| 描述 | 一句话风险陈述（"如果 X，则会 Y"） |
| 触发概率 | 高 / 中 / 低（基于历史数据 / 经验判断） |
| 影响等级 | 高 / 中 / 低（金额 / 用户数 / 故障时长） |
| 风险等级 | 概率 × 影响 → 红 / 黄 / 绿 |
| 缓解措施 | 当前方案中已采取的对策（指向 FT- / D- 编号） |
| 残余风险 | 缓解后仍存在的风险（是否可接受） |
| 责任人 | 风险跟踪 owner |

**示例**：

> **RISK-002**
> **类别**：技术
> **描述**：RocketMQ 顺序消费在单机降级或扩容时，可能短暂破坏顺序性。
> **触发概率**：低（年均 1–2 次扩缩容）
> **影响等级**：中（可能出现订单状态短暂错乱，业务需对账修复）
> **风险等级**：黄
> **缓解措施**：1) 启用对账任务每 10 分钟扫描订单状态一致性（FT-13）；2) 消费侧基于 orderId + version 做幂等（D-3）。
> **残余风险**：扩缩容期间 ≤ 5 分钟可能出现状态错乱，对账后可恢复，业务方接受。
> **责任人**：张三

### 7.2 待确认事项清单（TBD-）

依赖业务方、外部团队、平台方决策的事项，**必须**登记：

| 字段 | 说明 |
|---|---|
| 编号 | `TBD-001`、`TBD-002`… |
| 描述 | 待确认问题 |
| 阻塞内容 | 不确认会阻塞哪些章节或哪个里程碑 |
| 期望确认方 | 业务方 / DBA / SRE / 安全 / 合规 |
| 期望确认时间 | YYYY-MM-DD |
| 当前状态 | 待发起 / 已发起 / 已确认 / 已澄清 |
| 跟踪 owner | 谁负责推动 |

**示例**：

> **TBD-001**
> **描述**：是否允许跨地域同步用户敏感字段（手机号 / 身份证）？
> **阻塞内容**：第 6 章异地多活方案；第 4 章 D-5 数据分片决策。
> **期望确认方**：合规团队 + 业务方
> **期望确认时间**：2026-05-30
> **当前状态**：已发起（合规会议邮件 #12345）
> **跟踪 owner**：李四

### 7.3 待补全事项（TODO-，仅草案模式）

草案模式下允许暂时未展开的内容必须在此显式登记，不得隐藏：

| 编号 | 章节 | 待补全内容 | 期望补全时间 | Owner |
|---|---|---|---|---|
| TODO-001 | §3.1.2 | 缓存方案待与 SRE 确认是否使用 MemoryDB | 2026-05-25 | 张三 |

完整产出模式下不应出现 TODO-。

## 撰写规则

1. **风险不许写"已规避"**：所有风险都有残余概率，必须给出缓解措施 + 残余风险描述。
2. **不许把高风险藏成中风险**：等级判定需基于具体数字（如金额 / 影响用户数）而非主观意愿。
3. **TBD 必须有期望时间**：没有 deadline 的待确认事项 = 永远不会确认。
4. **责任人必须实名**：不能写"团队 / 项目组" — 没人负责 = 不会跟进。
5. **风险类别要平衡**：仅识别技术风险通常意味着没识全；业务 / 合规 / 团队风险也必须考虑。

## 风险识别参考清单（用于自检是否有遗漏）

逐条对照，命中的必须登记：

- [ ] **业务风险**：需求变更概率？是否有竞品上线压力？
- [ ] **数据风险**：是否涉及敏感数据？数据迁移是否有丢失风险？
- [ ] **性能风险**：QPS 预估是否准确？是否有热点 key？
- [ ] **可用性风险**：是否依赖单点？跨机房是否有真实演练？
- [ ] **安全风险**：鉴权是否完备？是否有越权风险？
- [ ] **合规风险**：是否触及个保 / 等保 / 跨境数据？
- [ ] **第三方依赖**：第三方 SLA 是否覆盖？降级方案是否充分？
- [ ] **团队风险**：核心成员是否单点？是否有培训成本？
- [ ] **时间风险**：里程碑是否可达？是否有外部依赖排期不确定？
- [ ] **预算风险**：机器 / 第三方 / 人力是否超预算？

## Checklist

- [ ] 每条风险包含 9 字段（编号 / 类别 / 描述 / 概率 / 影响 / 等级 / 缓解 / 残余 / 责任人）。
- [ ] 风险等级判定基于具体数字，不是主观判断。
- [ ] 已对照"风险识别参考清单"逐条自检。
- [ ] 每条 TBD 含 7 字段，**有期望确认时间** 与 **跟踪 owner**。
- [ ] 草案模式下的 TODO 已全部登记。
- [ ] 风险与缓解措施的引用编号（FT- / D- / FR- / NFR-）在前文有定义。

## 反模式

- ❌ "目前未发现风险" — 几乎不可能；通常意味着没认真识别。
- ❌ TBD 写"找时间确认" — 没 deadline 不会发生。
- ❌ 风险责任人写"团队" — 责任分散等于无责任。
- ❌ 风险等级靠感觉 — 必须是概率 × 影响的二维判定。
- ❌ 缓解措施与第 5 章脱钩 — 在 RISK 里说"会监控"但第 5 章没监控指标 = 假大空。
