# 反模式手册（撰写自检清单）

> 本文档列举技术方案产出过程中**必须避免**的高频反模式，并给出识别方法与修正建议。Step 4 自检阶段逐条比对。

## A. 画大饼式架构

### 识别
- 引入了当前需求并不需要的组件（如 NFR 中无多机房要求却加了"未来支持异地多活"的设计）。
- 任何用"为未来预留"作为核心理由的设计点。

### 危害
- 增加复杂度与运维成本，团队真实需求时无人维护。
- 评审被无关讨论占据时间。

### 修正
1. 删除当前不需要的组件。
2. 将"未来可能演进"放到附录"演进方向"，并标注前提条件（如"当 QPS > 10w 时考虑分片"）。
3. 不在主架构图中体现。

## B. 简历驱动设计（Resume-Driven Design）

### 识别
- 引入团队不熟悉的新技术，没有给出"和团队熟悉的成熟方案"对比。
- 决策理由出现"业界新趋势 / 大厂在用"等缺乏数据支撑的描述。

### 危害
- 上线后无人能维护；故障时手忙脚乱。
- 引入隐性成本（培训、踩坑、社区支持差）。

### 修正
1. 决策卡片中必须列出"团队熟悉的成熟方案"作为候选并对比。
2. 引入新技术必须给出"团队熟悉度提升计划"（培训、值班、回滚预案）。
3. 写明"如果新技术不行，回退到 X 方案的成本"。

## C. 隐藏复杂度（Hand-Waving）

### 识别
- 出现"后续再细化" / "暂不展开" / "由 XX 服务保证" 等绕过表述。
- 关键流程中存在含糊步骤，无法直接编码。

### 危害
- 评审时无法判断方案是否成立。
- 编码时才发现困难，方案被推翻。

### 修正
1. 把困难点直接展开到可评审粒度。
2. 实在不能在本方案展开的，写到第 7 章"风险与待确认"，给出 owner 和期望时间。
3. 任何"由 XX 保证"必须给出 XX 的具体 SLA 和接口契约。

## D. 缺少异常路径（Happy-Path-Only）

### 识别
- 流程描述里只有正常流转，没有超时、失败、重试、补偿、降级。
- 接口设计只列出 200 响应，没有 4xx / 5xx 错误码。

### 危害
- 真实生产环境异常路径占代码量 30%+，方案未覆盖等于实施时再设计。
- 测试用例无法对照方案编写。

### 修正
1. 每个流程的每一步都补足"异常 / 超时 / 重试 / 幂等"四项。
2. 接口设计补完错误码表。
3. 第 5 章故障矩阵覆盖核心异常场景。

## E. 概念混淆（Term Drift）

### 识别
- 同一对象出现多个名字（订单 / 单据 / order / 交易）。
- 状态、操作、字段名在不同章节不一致。
- 中英文混用且无映射。

### 危害
- 读者困惑，评审效率低下。
- 编码时字段命名混乱，文档与代码脱节。

### 修正
1. 全文统一一个名字，其它别名只在首次出现处用括号注明（如"订单（order）"）。
2. 必要的中英文映射就近括注；如确有大量缩写需登记，集中放到 `.qiqskills/backend-tech/<方案名>/notes.md`，**不在最终方案中单列术语表**。
3. 用工具 / Ctrl+F 全文搜索，机械替换。

## F. 需求脱钩（Design without Requirements）

### 识别
- 设计点找不到对应的 FR / NFR / ASMP 编号。
- 需求清单中有但设计未覆盖。

### 危害
- 设计了用不上的功能（浪费）；需要的功能没设计（漏洞）。

### 修正
1. 自查：每个设计点能否反向追溯到某条需求编号？不能就删除或补需求。
2. 自查：每条 FR/NFR 是否有对应的设计落实？没有的列入"待补"。
3. 必要时建立"需求-设计"双向矩阵。

## G. 决策黑箱（One-Choice Decision）

### 识别
- 关键技术决策只给出最终选择，没有候选方案与对比。
- 候选方案存在但描述敷衍（"方案 B 太重，不考虑"）。

### 危害
- 评审者无法判断决策是否合理。
- 方案被质疑时无据可援。

### 修正
1. 每个决策列 ≥ 2 个候选，每个候选给一句话描述。
2. 对比维度必须具体到数字或场景，禁止"性能更好"。
3. 理由都引用对比表中的格子。

## H. 数字模糊（Quantitative Hand-Waving）

### 识别
- 出现"大概"、"可能"、"应该差不多"、"高并发"、"大数据量"等无量化描述。
- QPS / 延迟 / 容量没有具体数。

### 危害
- 容量评估、性能预估无法验证。
- 上线后实际数据偏离预期，故障频发。

### 修正
1. 全文搜索"大概 / 可能 / 应该 / 大约 / 左右"，逐个替换为具体数字。
2. 数字给出来源（业务方 / 历史监控 / 类比系统 / 压测数据）。

## I. 单选题模式（Single-Solution Mindset）

### 识别
- 整个方案没有任何"取舍"语句。
- 不写被排除的选项，假装没有别的选择。

### 危害
- 后续遇到同类问题时复盘困难。
- 团队认知不一致。

### 修正
1. 关键节点显式记录"取舍"：选了 X 而不是 Y，因为 …。
2. 风险章节登记"被取舍掉的方案"在何种条件下需要重新评估。

## J. 监控告警仅在 5.4（Monitoring as Afterthought）

### 识别
- 把监控告警当成"事后再补"的事项。
- 未定义业务指标，只有系统指标（CPU / 内存）。

### 危害
- 上线后业务故障无法及时发现。
- 告警噪音大，真实告警被淹没。

### 修正
1. 业务指标（成功率 / 转化率 / 金额）必须先定义。
2. 阈值基于历史数据或压测，不是拍脑袋。
3. 告警分级 + 升级机制完备。

## K. 兼容性盲区（Breaking Change Blindspot）

### 识别
- 涉及接口或消息格式变更，未声明兼容期与下线流程。
- 未定义版本号策略。

### 危害
- 调用方被动适配，故障频发。
- 旧字段长期残留，技术债累积。

### 修正
1. 接口设计声明版本号策略（语义化版本 + URL/Header 版本号）。
2. 破坏性变更声明 ≥ 90 天兼容期 + 监控旧字段使用方 + 主动告警。
3. 第 6 章 / 第 7 章登记下线计划与 owner。

## L. 全栈"我"做（Org Blindspot）

### 识别
- 方案中所有事都"我们做"，未识别需要 DBA / SRE / 安全 / 数据 / 产品 配合的部分。

### 危害
- 上线时被外部排期阻塞。
- 跨团队沟通缺失。

### 修正
1. 第 1 章"组织依赖"列出所有需要外部团队配合的事项。
2. `.qiqskills/backend-tech/<方案名>/open-questions.md` 中 TBD 登记跨团队待办，给出期望确认时间与跟踪 owner（不写进最终方案文档）。

## M. 越权盲区（Authorization Blindspot）

### 识别
- 接口或查询条件未带"资源归属者"约束（如 `WHERE id = ?` 不带 `AND user_id = token.userId`）。
- 管理端接口没有独立的角色 / 权限校验。
- 仅靠前端隐藏菜单/按钮做"权限控制"，后端未独立校验。

### 危害
- 水平越权（IDOR）：用户访问到他人订单 / 资料 / 文件，触发严重数据泄露与合规事故。
- 垂直越权：普通用户调到管理员能力，可能造成大规模数据破坏。
- 功能越权：前端隐藏的危险接口被直接构造请求触发。

### 修正
1. 第 3 章接口契约必须显式声明"鉴权三要素"：身份、资源归属、角色 / 权限。
2. 服务端二次校验资源归属，不依赖 URL 参数 / 前端传入字段。
3. 第 5 章监控告警增加"鉴权失败计数"与"跨主体访问异常"指标；第 7 章登记越权类风险。
4. 上线前置条件加入"越权（水平 / 垂直 / 功能）渗透测试"。

## N. 漏洞盲区（Common Vulnerability Blindspot）

### 识别
- 用户输入直接进入 SQL / 命令 / 模板 / URL / 反序列化。
- 错误响应回显 SQL / 堆栈 / 内网 IP 等敏感信息。
- 依赖组件没有持续 CVE 扫描。
- 接口对外开放但无 WAF / 频控 / 异常监测。

### 危害
- 注入、RCE、SSRF、XSS、CSRF 等高危漏洞，单一漏洞可能导致整站沦陷。
- 已知 CVE 长期未修复，扫描器一查一个准。

### 修正
1. 第 3 章接口契约的"输入校验"列项强制白名单 + 强类型 + 长度上限。
2. 100% 参数化 SQL；禁用原生 Java 反序列化；SSRF 走域名 / IP 白名单；XSS 输出编码；CSRF token + SameSite。
3. 错误响应统一脱敏，不回显内部细节。
4. 第 5 章故障矩阵 FT-16 列出常见漏洞类与防护策略；第 7 章登记漏洞类风险；上线前置条件加入"OWASP Top 10 扫描 + 依赖 CVE 高危 ≤ 7 天修复"。

## O. 过程内容污染交付（Process Pollution）

### 识别
- 最终技术方案文档中出现完整的"质量自检表"、"待确认事项 TBD 表"、"待补全事项 TODO 表"，把过程产物当成交付内容。
- 第 1 章把需求采集模板的所有详细字段一字不漏地搬进交付文档。
- 第 6 章变成完整的运维操作手册 / Checklist 全集。

### 危害
- 交付文档变厚但信息密度下降，评审者迷失。
- 过程性内容（TBD / TODO / 自检细节）暴露在最终文档中，干扰决策视线。
- 同一信息在多处重复，修订时容易不一致。

### 修正
1. `.qiqskills/backend-tech/<方案名>/` 方案工作目录归档：`requirements.md`（详细需求）、`quality-check.md`（自检与门禁）、`open-questions.md`（TBD / TODO）、`notes.md`（其他过程产物）。
2. 最终方案中**仅保留**：业务背景摘要、FR / NFR 摘要表、约束与假设要点、架构与详细设计、关键决策、可用性与容错、部署运维要点、风险登记表。
3. 元信息头放门禁结论摘要 + TBD/TODO 条数摘要，正文不展开。
4. Step 7 一致性与篇幅微调阶段强制清理过程性内容残留。

## P. 附录膨胀（Appendix Bloat）

### 识别
- 附录章节越写越长：术语表数十条、参考资料链接全集、变更记录从 v1.0 累积到 v1.N 全列、演进方向罗列十几条。
- 附录中出现历次评审纪要全文、被取代的旧设计稿、长篇 SOP。
- 文档头部仍保留独立"术语表"小节（已被 E 项与本 skill 约束移除）。
- 文档头部生成"章节目录 / 本文目录 / Table of Contents"等导航段落，徒增篇幅。

### 危害
- 附录喧宾夺主，正文与索引比例失调。
- 历史信息堆积导致评审者抓不到本次重点。
- 同一信息在 `.qiqskills/backend-tech/<方案名>/notes.md` 与正文附录中两处维护，容易不一致。

### 修正
1. 最终方案不保留历史变更表；所有版本变更写入 `.qiqskills/backend-tech/<方案名>/notes.md` 的「历史变更归档」。
2. 演进方向、参考资料、旧设计、评审纪要等长尾信息归档到 `.qiqskills/backend-tech/<方案名>/notes.md`，附录只留必要索引。
3. 每个附录小节正文 ≤ 10 行；超过即重构为"概要 + 链接到 `.qiqskills/backend-tech/<方案名>/notes.md`"。
4. 术语表本身不在最终方案中保留（详见 E 项），P 项不重复处理。
5. **不生成章节目录（TOC）**：删除任何"本文目录 / Table of Contents"段落，章节定位靠标题层级。
6. Step 7 一致性与篇幅微调阶段强制清理。

## Q. 反馈修订发散（Feedback Scope Creep）

### 识别
- 用户只对已输出技术方案提出局部修改建议，却重写无关章节、扩展新目标或新增未被要求的技术选型。
- 修改时只处理了用户建议，没有复查由该修改引起的术语、编号、门禁状态、风险登记不一致。
- 修改完成后未完整重跑反模式自检、质量门禁和整体一致性检查。

### 危害
- 用户反馈被"扩写"成新的方案范围，导致评审焦点漂移。
- 无关改动引入新的不一致，增加二次返工成本。
- 阻断级问题被遗漏，方案看似已修订但仍不可评审。

### 修正
1. 将用户反馈拆成 `FB-001`、`FB-002`…，记录到 `.qiqskills/backend-tech/<方案名>/notes.md`，逐条标注处理结论。
2. 修改范围只覆盖用户反馈点、直接一致性影响和明确 Blocker；其他优化建议先登记到 notes，不进入本轮正文。
3. 修改后完整执行 Step 4 / Step 5 / Step 7；可自行修复的 Blocker 必须修复后再交付，依赖外部信息的登记为 ASMP / TBD。

## R. 超长不拆分（Oversized Single Document）

### 识别
- 最终技术方案文档超过 1000 行，但没有人工确认放开。
- 已超过 1000 行，却仍把多个相对独立功能模块写在同一篇文档中。
- 功能本身单一但实现链路很长时，没有按实现步骤 / 落地阶段拆分。
- 元信息 `状态` 行没有 `篇幅放开：确认人 + 日期 + 理由`，`.qiqskills/backend-tech/<方案名>/notes.md` 也没有确认依据。

### 危害
- 单篇文档过长，评审者难以聚焦核心链路和关键决策。
- 多模块内容耦合在一起，导致评审、排期、落地和回滚边界不清。
- 后续修订成本上升，容易出现章节间漂移和重复描述。

### 修正
1. 先执行 Step 7 篇幅收敛：合并重复段落、长尾枚举外移、操作手册化内容归档、精简第 1 / 6 章和附录。
2. 收敛后仍超过 1000 行且无人工确认时，必须拆分：优先按相对独立的功能模块拆分；功能单一时按实现步骤 / 落地阶段拆分。
3. 若人工明确确认允许单篇超过 1000 行，在元信息 `状态` 行标注 `篇幅放开：确认人 + 日期 + 理由`，并把确认依据写入 `.qiqskills/backend-tech/<方案名>/notes.md`。

## 自检流程

撰写完成后，对每条反模式（A–R）回答：

- 是否存在？
- 出现在哪里（章节定位）？
- 已修正还是登记到风险？

只有全部回答清楚，方可进入质量门禁。
