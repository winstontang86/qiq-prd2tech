# 质量门禁与自检规则

> 本文用于 Step 5 机械执行：把"方案是否达标"由主观判断转为字段化判定。完整产出模式必须通过门禁；草案模式仅输出自检结果，不阻断交付但需头部声明"草案 / 不可用于评审"。

## 二、Blocker 自动判定规则

满足任一条件 → 强制 Blocker，文档不可交付：

1. **业务目标无量化**：业务背景写"提升用户体验"等虚词。
2. **NFR 缺失关键维度**：可用性 / QPS / 延迟 / 数据一致性中任一未给具体数。
3. **关键决策候选 < 2**：存在"只列结论无对比"的决策。
4. **资金类需求无强一致设计**：涉及金额但未声明强一致 / 对账机制。
5. **接口无幂等 / 错误码缺失**：写接口的同时不提幂等与错误码。
6. **流程无异常路径**：核心流程仅 happy path，无超时 / 失败处理。
7. **故障矩阵未覆盖 SLA 对应场景**：NFR 写 99.95%，但故障矩阵中机房故障未涉及。
8. **RPO/RTO 与 NFR 矛盾**：NFR 要求 RPO=0，但备份策略给出"每小时一次全量备份"。
9. **破坏性变更无兼容期**：方案涉及破坏性变更但未声明兼容期与下线流程。
10. **草案模式但 ASMP 未登记**：草案缺信息却不显式登记假设。

## 三、门禁判定（完整产出模式）

按下表机械推导：

| 条件 | 结果 |
|---|---|
| 任一 Blocker 项失败 | **不通过**：文档不可用于评审，必须修复后重审。 |
| 0 Blocker，但 High ≥ 3 | **有条件通过**：必须在文档头部列出 High 项整改计划与 deadline；评审会上首要议题。 |
| 0 Blocker，High ≤ 2，Medium 不限 | **通过**：可进入正式评审。 |

## 四、门禁判定（草案模式）

不输出"通过 / 不通过"，但**必须输出**：

- 草案声明（"本文档为草案，置信度 X，不可用于上线评审"）。
- 已登记的 ASMP- 与 TODO- 清单。
- 完整产出前还需补完的章节列表。
- 期望补完时间与 owner。

## 五、文档头部"质量自检"小节模板

每份方案末尾必须输出：

```markdown
## 质量自检

| 项 | 编号 | 状态 | 备注 |
|---|---|---|---|
| 业务目标量化 | Q-2 | ✅ | |
| FR 字段齐全 | Q-3 | ✅ | 共 12 条 |
| NFR 字段齐全 | Q-4 | ✅ | 共 9 条 |
| ASMP 登记 | Q-5 | ⚠️ | 3 条待业务方确认 |
| 架构图 | Q-6 | ✅ | |
| 关键决策 | Q-14 | ✅ | 5 个决策，各 ≥ 2 候选 |
| 故障矩阵 | Q-15 | ✅ | 9 类全覆盖 |
| ...（逐条列出 Q-1 至 Q-26） | | | |

**Blocker 项**：0
**High 项**：1（Q-12 流程缺幂等说明，已计划补完，2026-05-22）
**Medium 项**：2

- A 画大饼：未发现
- B 简历驱动：未发现
- C 隐藏复杂度：1 处（§3.3 异常补偿任务，已展开）
- ...（逐条）
```

## 六、修订规则

每次修订：

1. **版本号 +1**（语义化：大改动主版本，小改动子版本，错别字补丁版本）。
2. 文档头部"变更摘要"补一行：
   ```
   v1.2 (2026-05-18) — 张三：补充 D-3 缓存击穿防护；修正 NFR-005 数字；新增 RISK-007。
   ```
3. 重新执行本文档的自检，更新"质量自检"小节。

## 七、强制约束

- **不允许"看情况" / "视情况"**：所有判定必须基于上方表格的客观字段。
- **Blocker 不可豁免**：等同于"文档未达标"。
- **草案模式不可越级**：声明草案的方案不得在评审会上作为决策依据，仅供讨论。
- **修复后必须重审**：任一 Blocker / High 整改后，需重新跑完 Q-1 至 Q-26 的检查并更新自检小节。
